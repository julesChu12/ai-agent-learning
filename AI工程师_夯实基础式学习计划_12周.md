### Week 1：AI基础 + LLM机制

**目标：** 理解LLM基本原理，为后续学习打基础

#### 1.1 LLM基础概念

**核心知识点：**

```
LLM (Large Language Model) = 大语言模型

关键概念：
├── Token：文本最小单元（中文约2-3字=1 token）
├── Context Window：模型能处理的上下文长度
├── Temperature：控制随机性（0=确定，1=随机）
├── Top-p：核采样，控制输出多样性
└── Streaming：流式输出，边生成边返回
```

**图解LLM工作流程：**
```
用户输入 → Tokenize(分词) → Embedding → Transformer → Output Prob → De-tokenize → 文本
                                                     ↑
                                              预训练权重
```

#### 1.2 主流LLM API对比

| 模型 | 特点 | 适用场景 |
|------|------|---------|
| GPT-4/3.5 | 通用能力强 | 复杂推理、多轮对话 |
| Claude | 长上下文、安全 | 长文档分析 |
| 通义千问 | 中文强、开源 | 国内业务 |
| 文心一言 | 百度生态 | 企业内部 |
| Llama | 开源可私有 | 私有化部署 |

**API调用示例（Python）：**
```python
from openai import OpenAI

client = OpenAI(api_key="your-api-key")

response = client.chat.completions.create(
    model="gpt-3.5-turbo",
    messages=[
        {"role": "system", "content": "你是一个助手"},
        {"role": "user", "content": "解释什么是Token"}
    ],
    temperature=0.7,
    stream=False  # 设为True可流式输出
)

print(response.choices[0].message.content)
```

**流式输出示例（Go）：**
```go
// Go + goroutine + channel 处理LLM流式响应
func streamChat(ctx context.Context, prompt string) (<-chan string, <-chan error) {
    respCh := make(chan string)
    errCh := make(chan error, 1)

    go func() {
        defer close(respCh)
        defer close(errCh)

        req := &Request{Prompt: prompt}
        stream, err := client.CreateChatCompletionStream(ctx, req)
        if err != nil {
            errCh <- err
            return
        }
        defer stream.Close()

        for {
            select {
            case <-ctx.Done():
                return
            default:
                resp, err := stream.Recv()
                if errors.Is(err, io.EOF) {
                    return
                }
                if err != nil {
                    errCh <- err
                    return
                }
                respCh <- resp.Choices[0].Delta.Content
            }
        }
    }()

    return respCh, errCh
}
```

### Week 2：RAG核心概念 + 全链路

**目标：** 理解RAG为什么存在，以及完整流程

#### 2.1 为什么需要RAG？

**LLM的三大局限：**
```
1. 知识截止日期（不知道最新信息）
2. 幻觉（一本正经地胡说八道）
3. 长文档处理受限（Context Window有限）

RAG = Retrieval Augmented Generation（检索增强生成）
作用：让LLM"查资料"后再回答，而不是凭空编造
```

**RAG vs Fine-tuning：**

| 对比 | RAG | Fine-tuning |
|------|-----|------------|
| 成本 | 低 | 高（训练成本） |
| 时效性 | 高（实时检索） | 低（需重新训练） |
| 适合场景 | 知识库问答 | 风格/领域适应 |
| 幻觉消除 | 强 | 中等 |

#### 2.2 RAG完整流程

```
┌──────────────────────────────────────────────────────────────┐
│                        RAG 流程                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【文档侧】                    【查询侧】                      │
│                                                              │
│  文档 → 分块 → Embedding → 存入向量库                        │
│                                      ↑                        │
│                                      │                       │
│  ┌───────────────────────────────────┘                       │
│  ↓                                                           │
│  用户问题 → Embedding → 检索向量库 → 获取Top-K相关块           │
│                                    ↓                         │
│                            组装Prompt                         │
│                                    ↓                         │
│                              LLM 生成                        │
│                                    ↓                         │
│                               最终答案                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

#### 2.3 RAG评估指标

**核心指标：**

| 指标 | 含义 | 理想值 |
|------|------|--------|
| 召回率（Recall） | 相关文档被检索到的比例 | 越高越好 |
| 精确率（Precision） | 检索结果中相关的比例 | 越高越好 |
| MRR | 第一个相关结果的位置 | 越靠前越好 |
| Hit Rate | 至少检索到一个相关文档的比例 | 越高越好 |

### Week 3：Embedding + 向量数据库

**目标：** 理解Embedding原理，掌握向量数据库使用

#### 3.1 Embedding是什么？

**定义：** 将文本映射为稠密向量（通常768维、1024维、1536维）

**类比理解：**
```
"苹果" → [0.23, -0.45, 0.78, ...]
"香蕉" → [0.25, -0.43, 0.75, ...]
"手机" → [-0.12, 0.56, -0.89, ...]

苹果和香蕉向量距离近（水果同类）
苹果和手机向量距离远（不同类别）
```

**主流Embedding模型：**

| 模型 | 维度 | 特点 | 适用语言 |
|------|------|------|---------|
| BGE | 1024 | 追平OpenAI, 可商用 | 中英 |
| M3E | 1536 | 高效, 支持中文 | 中英 |
| Text2Vec | 768 | 经典, 开源 | 中英 |
| OpenAI-ada-002 | 1536 | 最强, 付费 | 通用 |

### Week 5：LangChain 基础

**目标：** 掌握LangChain核心组件，能构建简单Chain

#### 5.1 LangChain是什么？

```
LangChain = LLM应用的"脚手架"框架

核心价值：
├── 统一接口：不用关心底层LLM差异
├── 组件化：Prompt、Model、OutputParser可组合
├── Chain：把多个步骤串成流水线
└── Agent：自动决策+工具调用
```

#### 5.2 核心组件

**Model（模型）：**
```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="gpt-3.5-turbo",
    temperature=0.7,
    api_key="your-key"
)

response = llm.invoke("解释什么是LangChain")
```

**Prompt Template：**
```python
from langchain_core.prompts import PromptTemplate

template = PromptTemplate.from_template(
    "请用{language}翻译以下文本：{text}"
)

prompt = template.invoke({
    "language": "英文",
    "text": "LangChain让LLM开发变简单"
})
```

**Output Parser：**
```python
from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel

class Answer(BaseModel):
    translation: str
    original_lang: str

parser = JsonOutputParser(pydantic_object=Answer)

chain = template | llm | parser
result = chain.invoke({"text": "你好"})
# result = {"translation": "Hello", "original_lang": "中文"}
```

#### 5.3 简单Chain示例

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 模板

    "用一句话解释{concept}，举例说明"

# 输出解析

# 串成Chain

# 执行
result = chain.invoke({"concept": "RAG"})
```

### Week 7：LangGraph + Agent循环

**目标：** 掌握LangGraph，能实现复杂Agent循环

#### 7.1 为什么需要LangGraph？

```
LangChain Chain：线性流程 A → B → C → D
LangGraph：可以循环，可以条件分支

适用场景：
├── Agent：需要反复思考+行动
├── 复杂工作流：多分支、循环
└── 状态管理：需要保存中间状态
```

#### 7.2 StateGraph核心概念

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

# 1. 定义State
class AgentState(TypedDict):
    messages: List[str]      # 对话历史
    next_action: str         # 下一步动作
    iteration: int            # 迭代次数

# 2. 定义Node（处理函数）
def think_node(state: AgentState) -> AgentState:
    # LLM决定下一步
    action = llm_think(state["messages"])
    return {"next_action": action, "iteration": state["iteration"] + 1}

def action_node(state: AgentState) -> AgentState:
    # 执行动作
    result = execute_tool(state["next_action"])
    return {"messages": state["messages"] + [result]}

# 3. 定义边（转换规则）
def should_continue(state: AgentState) -> str:
    if state["iteration"] > 10:
        return "end"
    return "think"

# 4. 构建图
graph = StateGraph(AgentState)
graph.add_node("think", think_node)
graph.add_node("action", action_node)

graph.add_edge("__start__", "think")
graph.add_conditional_edges("think", should_continue, {
    "continue": "action",
    "end": END
})
graph.add_edge("action", "think")

# 5. 编译
app = graph.compile()
```

#### 7.3 ReAct Agent完整实现

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class ReActState(TypedDict):
    query: str
    thought: str
    action: str
    observation: str
    response: str

def reason_node(state: ReActState) -> ReActState:
    """Think: LLM推理下一步"""
    prompt = f"""
Query: {state['query']}
Previous Thoughts: {state.get('thought', '')}
Previous Actions: {state.get('action', '')}
Previous Observations: {state.get('observation', '')}

根据上下文，决定下一步：
1. 如果需要调用工具，格式：Action: tool_name(arg)
2. 如果已得到答案，格式：Final: <answer>

你的决定："""
    
    decision = llm.invoke(prompt)
    if decision.startswith("Final:"):
        return {"response": decision.replace("Final:", "").strip()}
    else:
        return {"thought": decision}

def action_node(state: ReActState) -> ReActState:
    """执行工具调用"""
    if state["action"].startswith("search_"):
        query = extract_query(state["action"])
        result = search_tool(query)
        return {"observation": str(result)}
    return {"observation": "unknown action"}

# 构建图
graph = StateGraph(ReActState)
graph.add_node("reason", reason_node)
graph.add_node("action", action_node)
graph.add_conditional_edges("reason", lambda s: "end" if s.get("response") else "action")
graph.add_edge("action", "reason")

app = graph.compile()
```

```python
# ⚠️ Q5知识点：幂等性 = 唯一请求ID追踪

class IdempotentMCPTool:
    """保证MCP Tool的幂等性"""
    
    def __init__(self):
        self.processed: Dict[str, dict] = {}
        # ↑ 存储已处理的请求ID和结果
    
    def invoke(self, request_id: str, params: dict) -> dict:
        """
        1. 检查request_id是否已处理
        2. 如果是，返回缓存结果
        3. 如果否，执行处理并缓存结果
        """
        # Step 1: 检查
        if request_id in self.processed:
            return {
                "status": "duplicate",
                "cached": True,
                "result": self.processed[request_id]
            }
        
        # Step 2: 首次处理
        result = self._do_process(params)
        
        # Step 3: 缓存
        self.processed[request_id] = result
        
        return {
            "status": "processed",
            "cached": False,
            "result": result
        }
    
    def _do_process(self, params: dict) -> dict:
        """实际业务逻辑"""
        # ...
        pass
```

#### 11.2 千万级架构关键组件

**1. 限流熔断：**
```python
# 常见方案：令牌桶/滑动窗口
from ratelimit import limits, sleep_and_retry

@sleep_and_retry
@limits(calls=100, period=60)  # 60秒内最多100次
def call_llm(prompt):
    return openai.ChatCompletion.create(prompt=prompt)
```

**2. 缓存策略：**
```
┌─────────────────────────────────────────┐
│           LLM 缓存层                     │
├─────────────────────────────────────────┤
│ Request Hash → 相同Prompt → 返回缓存    │
│                                         │
│ 缓存维度：                               │
│ ├── Prompt完全匹配                       │
│ ├── Semantic相似（向量距离）             │
│ └── 混合策略                            │
└─────────────────────────────────────────┘
```

**3. 多级缓存：**
```
请求 → Redis缓存 → LLM → 响应
              ↓命中
           返回缓存
```

#### 11.3 Go工程师的优势

**Go在AI后端的优势：**
```
1. 高并发：goroutine处理大量并发LLM调用
2. 低延迟：网络I/O优化
3. 部署简单：静态编译，单二进制
4. 可观测：丰富的监控生态（Prometheus/Grafana）
```

**示例：Go调用LLM + 并发控制：**
```go
func main() {
    // 使用semaphore控制并发
    sem := make(chan struct{}, 100) // 最多100并发
    
    var wg sync.WaitGroup
    for _, prompt := range prompts {
        wg.Add(1)
        sem <- struct{}{}
        go func(p string) {
            defer wg.Done()
            defer func() { <-sem }()
            
            resp, err := callLLM(p)
            // 处理响应
        }(prompt)
    }
    wg.Wait()
}
```

#### 11.6 生产踩坑

| 坑 | 原因 | 解决方案 |
|----|------|---------|
| **LLM超时** | 模型响应慢/排队过长 | 熔断+超时降级+异步队列 |
| **成本爆炸** | 没限制Token用量 | 按用户/接口限Token+配额 |
| **缓存失效** | prompt轻微变化导致缓存miss | Semantic Cache（向量相似度） |
| **监控缺失** | 不知道模型延迟分布 | LangSmith/Prometheus+自定义指标 |

#### 12.4 项目面试表达

**面试官问："你这个项目遇到了什么难点？怎么解决的？"**

参考回答：
> "我在实现RAG Pipeline时，主要遇到两个问题：
> 1. 检索召回率低——通过对比BGE/M3E/Text2Vec，最终选择BGE作为Embedding模型，召回率从65%提升到82%；
